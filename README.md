### <1> Git

**1. 버전 관리**
* 버전 관리엔 '형상 관리'와 '버전 관리'가 있다.
* 형상 관리가 소프트웨어 시스템의 변화를 체계적으로 관리하는 모든 프로세스라면, 버전 관리는 형상 관리의 활동 중 하나로 소스 코드 변경 이력을 체계적으로 기록하며 과거 버전과 비교하거나 되돌릴 수 있도록 관리하는 것이다.
* Git이란, 2026년 기준 업계 표준으로 자리 잡은 버전 관리 시스템이다.
* **Fun Fact)** 리눅스 커널을 개발한 Linus Torvalds가 Git을 만들었다.
* **Git GUI에 대해**: Git은 CLI도 있지만, GUI도 있다. 즉, 명령어를 외워도 좋지만 툴을 사용해 응용을 할 줄 알고 상관없다는 것.

**2. Git Data Flow**
* 작업 디렉토리 ---> 스테이징 영역 ---> 로컬 저장소 ---> 원격 저장소
    * `add` / `commit` / `push`
* 작업 디렉토리 <--- 스테이징 영역 <--- 로컬 저장소 <--- 원격 저장소
    * `restore` / `reset` / `pull`

**3. 정방향 흐름**
* **`commit`**: 변경사항을 모아서 하나의 '버전'을 만들어 레포지토리에 기록한다. 폴더의 모든 파일을 복사하는 것이 아니라, 이전 버전과의 차이점(어느 파일의 몇 번째 줄을 추가/삭제했는지)만 저장한다. 메모리를 효율적으로 사용할 수 있고, 수정 내역을 추적할 수 있다.
    * **명령어**: `git commit -m <msg>`
* **`add`**: 어떤 변경사항을 커밋할지 선택한다. 여러 파일을 수정했는데, 그 변경사항 중 일부만 커밋하고 싶을 때 유용하다.
    * **참고)** 소스트리는 변경사항을 add할 때 라인별로 관리할 수 있다.
    * **명령어**: (특정 파일일 경우) `git add <pathspec>` (전부) `git add .`
* **`.gitignore`**: .gitignore에 파일을 적으면, Git은 그 파일을 추적하지 않는다. add하려는데 파일이 너무 많을 때 사용한다! 이걸 작년에 알았더라면...

**4. 역방향 흐름**
* **`reset`**: 특정 시점의 커밋(Known state)으로 현재 상태를 되돌린다.
    * **명령어**: `git reset <mode> <commit>`
    * **세 가지 모드**: 작업 디렉토리를 해당 커밋으로 바꾸고,
        * `--soft`: 차이점을 스테이징 영역에 둔다. (방금 한 커밋만 취소하고 바로 다시 커밋하고 싶을 때)
        * `--mixed`: 차이점을 작업 디렉토리에 둔다. (스테이징까지 다 취소하고 코드를 다시 고치고 싶을 때)
        * `--hard`: 차이점을 삭제한다. (실수한 내용 완벽히 없애고 싶을 때, 작업 내용 날아가니 주의!)
    * **돌아갈 지점을 지정할 때**:
        * `HEAD~n`: 현 커밋으로부터 n번째 이전 커밋으로 돌아간다.
        * `커밋 해시`: 레포에서 커밋을 식별할 수 있는 문자열(예: a1b2c3d)을 직접 입력할 수도 있다.
* **`restore`**: 스테이징 취소 / 변경사항 삭제
    * 파일을 수정하고 add 명령어로 스테이징 영역에 올렸는데, 커밋에 포함시키고 싶지 않을 경우에 스테이징을 취소!
        * **명령어**: (특정 파일일 경우) `git restore --staged <pathspec>` (전부) `git restore --staged .`
    * 파일을 수정했는데 마음에 들지 않아, 아예 마지막으로 커밋했던 상태로 파일을 되돌리고 싶을 때 변경사항 삭제!
        * **명령어**: (특정 파일) `git restore <pathspec>` (전부) `git restore .`

**5. Branch**
* **`branch`**: 소스 코드의 분기점을 만든다. 어떤 식으로 개발해나갈지 여러 방식을 생각할 때, 팀 프로젝트에서 여러 명이 동시에 작업할 때 유용하다.
    * **명령어**: `git branch <branch-name>`
* **`checkout`**: 다른 브랜치로 전환한다.
    * **명령어**: `git checkout <branch-name>`
* **`merge`**: 변경사항을 병합한다. 병렬 작업의 변경사항을 자동으로 합칠 수 있다.
    * **명령어**: `git merge <branch-name>`
* **`conflict`**: Git이 변경사항을 자동으로 합쳐주고 싶은데, 애매한 게 있어서 사용자에게 직접 병합하라고 요청하는 경우이다. 동시에 같은 브랜치를 수정했을 때 발생할 수 있는 부분이다. 걱정하지 말 것!

**6. Remote Repository**
* 원격 저장소는 로컬 저장소와 달리 다른 사람과 공유된다. 로컬 저장소는 고유하므로 개인 작업을 관리할 수 있다! 로컬 저장소의 내용들 중 필요한 것만 선택적으로 원격 저장소에 공유한다.
* **`push`**: 로컬 저장소의 커밋 이력을 원격 저장소에 업로드한다.
    * **명령어**: `git push origin <branch-name>`
* **`pull`**: 원격 저장소의 커밋 이력을 가져와서, 내 로컬 저장소에 반영한다.
    * **명령어**: (현재 브랜치) `git pull` (특정 브랜치) `git pull origin <branch-name>`
* **`clone`**: 레포지토리를 로컬로 복제한다. 원격 저장소를 "최초로" 복제할 때만 필요하다!


### <2> GitHub

**1. Git 호스팅 서비스**
* Git Repository를 서버에 두고, 인터넷을 통해 소스 코드를 관리할 수 있게 해주는 서비스
* 단순히 코드를 저장하는 것을 넘어, 팀원들과 협업하고 프로젝트를 배포하는 데 필수적인 다양한 기능을 제공한다.
* GitHub, GitLab, BitBucket 이 있다!
* 그 중에서도 GitHub은 Git 호스팅 서비스 중에서 세계에서 가장 큰 규모와 영향력을 가진 플랫폼이다. 전 세계 개발자들이 소스 코드를 올리고, 서로의 코드를 공유하며 협업하는 '개발자들의 SNS'라고도 불린다.

**2. Organization**
* GitHub의 Organization은 여러 사람이 공동으로 프로젝트를 관리하고 협업할 수 있도록 설계된 그룹 계정이다.
* 리포지토리를 함께 관리하고 공개/비공개 설정을 할 수 있다. 멤버 초대하는 것 잊지 말기!

**3. Repository**
* 프로젝트의 모든 파일과 그 파일들이 변화해 온 전체 히스토리를 담고 있는 가상의 보관함이다.

**4. Pull Request**
* 한 브랜치에서 다른 브랜치로 Merge를 요청하는 것이다.
* '네가 나를 pull 해줘라.'라고 요청하는 것이라 보면 된다. 그래서 pull

## 🔑 Key
**1. 대표님의 꿀팁!**
* **커밋과 PR의 단위를 정해라!**
    * **커밋의 단위**: reset하기 좋은, 작업 등(예: import문 수정)
    * **PR의 단위**: 코드 리뷰하기 좋은, 기능 등(예: 버튼 컴포넌트, 게시물 추가 API)
* **Repository 설정을 활용해라!**
    * `Automatically delete head branches` / `branch ruleset`
* **컨벤션을 정해라!**
    * 클린코드를 위해 필수적이다!
    * `Git Commit Message Convention` / `Pull Request Convention`
* **README를 작성해라!**
    * 레포지토리 설명서 / 마크다운 문법으로
    * 나중에 어떤 프로젝트를 했었는지 기억나지 않으므로, 그때그때 작성하기!
